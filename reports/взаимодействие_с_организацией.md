# Отчёт о взаимодействии с организацией-партнёром

## Общая информация

В рамках проектной (учебной) практики была выполнена производственная деятельность в международной fintech-компании, занимающейся разработкой высоконагруженных backend-систем, ML-инфраструктуры и сервисов real-time анализа пользовательской активности.

Практика проходила в формате участия в разработке production-grade сервисов, развёрнутых в Kubernetes-инфраструктуре компании и интегрированных в существующую ML ecosystem.

В процессе работы использовался следующий технологический стек:

- Python 3.13
- FastAPI
- ClickHouse
- MongoDB
- S3 Object Storage
- Kubernetes
- Argo Workflows
- ArgoCD
- GitLab CI/CD
- Docker
- ONNX Runtime
- Prometheus
- Grafana
- Git
- Linux

---

# Цель выполненной задачи

Основной задачей являлась разработка системы непрерывного переобучения ML-моделей классификации пользовательской активности с автоматическим rollout новой версии модели в production runtime без рестарта inference-сервисов.

Система должна была обеспечивать:

- автоматическую подготовку обучающих данных;
- регулярное переобучение ML-моделей;
- публикацию новых версий моделей;
- realtime обновление inference runtime;
- отказоустойчивость;
- observability;
- интеграцию с существующей Kubernetes-инфраструктурой компании.

---

# Архитектура системы

Система была реализована как распределённый MLOps pipeline, состоящий из нескольких независимых компонентов.

Основные компоненты системы:

1. `Snapshots Workflow`
2. `ML Training Workflow`
3. `Inference Service`
4. `S3 Object Storage`
5. `Notification Layer`
6. `ONNX Runtime`

Архитектура решения построена по принципу weak coupling между компонентами, что позволило независимо масштабировать отдельные части системы и минимизировать downtime при обновлениях.

---

# Snapshots Workflow

## Назначение компонента

`Snapshots Workflow` отвечает за формирование обучающих снапшотов и подготовку датасетов для последующего ML-обучения.

Workflow запускается автоматически каждый час через Argo Workflows CronWorkflow внутри ML Kubernetes-кластера.

---

## Источники данных

При формировании снапшотов использовались данные из аналитического ClickHouse-кластера:

- behavioral telemetry;
- captcha events;
- sms verification events;
- feature vectors;
- components_data;
- captcha_moves.

---

## Обработка данных

Во время обработки данные автоматически распределялись по нескольким категориям:

### Regular

Корректно классифицированные события.

### FP (False Positive)

Легитимные пользователи, ошибочно классифицированные моделью как боты.

### FN (False Negative)

Боты, ошибочно классифицированные как легитимные пользователи.

---

## Формирование снапшотов

После preprocessing workflow формировал ML snapshots, содержащие:

- feature vectors;
- labels;
- metadata;
- auxiliary telemetry.

Сформированные снапшоты сериализовывались и публиковались в S3 Object Storage.

---

## Deployment

Workflow разворачивался через GitOps pipeline:

1. Push изменений в GitLab repository;
2. Сборка Docker image через GitLab CI;
3. Публикация image в internal container registry;
4. Автоматическая синхронизация manifests через ArgoCD;
5. Deployment в Kubernetes cluster.

Для orchestration использовались Argo Workflows CronWorkflow resources.

---

# ML Training Workflow

## Назначение

`ML Training Workflow` отвечает за автоматическое ежедневное переобучение ML-модели.

Workflow запускается один раз в сутки в ML Kubernetes-кластере.

---

## Подготовка обучающей выборки

Workflow выполняет загрузку снапшотов за последние 90 дней из S3 Object Storage.

Для обучения используются данные следующих категорий:

- regular;
- fp;
- fn.

Использование FP/FN категорий позволяет дополнительно обучать модель на ошибках предыдущих inference cycles и улучшать качество классификации.

---

## Preprocessing

Перед обучением выполняются:

- validation;
- normalization;
- balancing;
- corrupted samples filtering;
- feature compatibility checks.

---

## Обучение модели

После подготовки датасета запускается процесс ML training.

Во время обучения вычисляются:

- accuracy metrics;
- validation metrics;
- model evaluation metrics;
- comparison metrics относительно предыдущей production-модели.

---

## Экспорт модели

После завершения обучения модель экспортируется в формат ONNX.

Использование ONNX Runtime позволяет:

- уменьшить latency inference;
- стандартизировать execution environment;
- повысить portability;
- ускорить runtime execution.

---

## Публикация модели

Новая версия модели публикуется в S3 Object Storage.

Каждая версия содержит:

- version metadata;
- model artifacts;
- export information;
- training metadata.

# Notification Layer

## Проблема межкластерного взаимодействия

Во время проектирования системы рассматривался вариант использования KeyDB Pub/Sub для realtime notification delivery.

Однако inference-сервисы и ML workflows были развёрнуты в разных Kubernetes-кластерах:

- ML workflows размещались внутри отдельного ML cluster;
- inference services размещались в production services cluster.

При этом KeyDB внутри инфраструктуры компании разворачивался через Kubernetes Operator и был доступен только внутри собственного Kubernetes network boundary.

Из-за этого direct Pub/Sub communication между кластерами становился инфраструктурно сложным и требовал дополнительного inter-cluster networking.

---

## Реализованное решение

В качестве notification layer была использована MongoDB, развёрнутая на bare-metal infrastructure и доступная из обоих кластеров.

После публикации новой модели `ML Training Workflow` создаёт notification document в MongoDB collection.

---

## Механизм realtime обновления

Внутри `Inference Service` был реализован background worker, выполняющий polling MongoDB collection каждые 30 секунд.

При обнаружении новой notification:

1. сервис получает metadata новой модели;
2. скачивает ONNX artifact из S3;
3. выполняет model validation;
4. atomically переключает runtime на новую версию модели;
5. освобождает старую версию runtime model.

---

## Причины выбора polling architecture

Polling-based architecture была выбрана как компромисс между:

- инфраструктурной простотой;
- отказоустойчивостью;
- независимостью кластеров;
- минимальным operational overhead.

Использование MongoDB позволило избежать сложной inter-cluster service mesh integration и обеспечить стабильную доставку notification events.

# Inference Service

## Назначение

`Inference Service` представляет собой production inference backend, выполняющий ML-классификацию пользовательской активности в режиме реального времени.

Сервис развёрнут в Kubernetes и обрабатывает production traffic.

---

## Загрузка модели

При старте сервиса выполняется:

1. получение актуальной версии модели из S3;
2. инициализация ONNX Runtime;
3. регистрация модели внутри inference pipeline.

---

## Runtime model reload

Ключевой особенностью системы является возможность обновления модели без рестарта сервиса.

После получения notification service:

1. загружает новую модель;
2. создаёт новый ONNX Runtime session;
3. atomically переключает active runtime pointer;
4. продолжает обработку запросов без interruption.

Это позволило исключить:

- rolling restart deployment;
- downtime;
- interruption active requests;
- traffic degradation during rollout.

# Kubernetes Infrastructure

## Containerization

Все компоненты системы контейнеризированы через Docker.

Для каждого сервиса собирались отдельные production images через GitLab CI/CD pipeline.

---

## GitOps Deployment

Deployment инфраструктуры был построен по GitOps-модели.

Использовались:

- GitLab CI/CD;
- ArgoCD;
- Kubernetes manifests;
- Helm templates.

После merge изменений ArgoCD автоматически синхронизировал desired state с Kubernetes cluster state.

---

## Autoscaling

Для inference-сервисов был настроен Horizontal Pod Autoscaler (HPA).

Autoscaling выполнялся на основании:

- CPU utilization;
- memory consumption;
- request throughput.

При увеличении нагрузки Kubernetes автоматически увеличивал количество pod replicas.

При снижении нагрузки происходило автоматическое scale down.

---

## Resource Management

Для сервисов были настроены:

- resource requests;
- resource limits;
- liveness probes;
- readiness probes.

Это обеспечивало:

- корректный scheduling;
- стабильность runtime;
- отказоустойчивость;
- корректное распределение ресурсов внутри Kubernetes cluster.
